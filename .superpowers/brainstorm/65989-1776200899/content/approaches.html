Three approaches to building this

Same visual design, same workflow — they differ in how much you customize vs. inherit from the starter template.

APPROACH A
website/ (existing Astro starter)
├── restyle global.css → serif + red accent
├── rewrite index.astro → your homepage
├── add remark-math + rehype-katex
├── replace header icons → LinkedIn/X/email
└── delete sample blog posts, add yours
Builds on what's already there

A — Restyle the starter (recommended)

Keep the Astro blog starter as-is, restyle it to match Literary Minimal, add LaTeX. Fastest path. The starter already has MDX, content collections, RSS, and a solid blog layout. No reason to rebuild what works.

Pros

  • Done fastest
  • Proven blog infrastructure
  • Easy to maintain

Cons

  • Inherits some starter opinions on structure
APPROACH B
website/ (fresh Astro + manual setup)
├── design system in global.css from scratch
├── layouts/Base.astro + layouts/BlogPost.astro
├── content collections for blog
└── add math, syntax highlight, image support
Cleaner starting point, more work upfront

B — Build from a minimal Astro base

Start from a blank Astro project and build only what you need. More upfront work but the codebase is yours from line 1 — no inherited opinions to work around.

Pros

  • Clean, no legacy starter code
  • Full control over structure

Cons

  • More setup
  • Same end result as A
APPROACH C
website/ (Astro + Tailwind Typography)
├── @tailwindcss/typography for prose styling
├── custom Tailwind theme → serif + red
└── math, code blocks via plugins
Utility-first, great prose defaults

C — Astro + Tailwind Typography

Add Tailwind and its Typography plugin. Gives you beautiful prose defaults out of the box — headings, code blocks, blockquotes all styled automatically in your blog posts.

Pros

  • Best prose defaults
  • Easy to tweak

Cons

  • Tailwind is more complexity than you need
  • Overkill for a minimal site
